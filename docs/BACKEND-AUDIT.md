# Backend Audit — Broken & Missing Flows

_Generated by static analysis: every `@SubscribeMessage` handler in the realtime
gateway diffed against every socket event the web/mobile/rider clients emit,
plus a scan of services for stubs and of the portals for mock write-actions._

## Summary

| Area | Status |
|------|--------|
| Realtime handlers | **97** across 17 modules, **30 entities** |
| Frontend read events with no handler | **0** — every read the UI performs is backed |
| Backend services with stub / TODO / not-implemented logic | **0** (payment gateway is an intentional mock) |
| **Genuinely-missing handlers** (capabilities the UI needs, no backend) | **7** → implementing now |
| **Broken flows** (UI write-actions that mutate locally instead of calling an existing handler) | ~40 across portals |

The backend is functionally complete for reads. The two real gaps are:

1. **Missing handlers** — a handful of user capabilities have no server endpoint.
2. **Unwired write-actions** — ~54 handlers exist but the UI never calls them;
   buttons like "Approve seller", "Resolve dispute", "Build batch" run a local
   `setState` + toast instead of persisting. The handler is there; the click is
   not wired to it.

## 1. Genuinely-missing backend (being implemented)

| Handler | Purpose | Owner |
|---------|---------|-------|
| `orders:cancel` | Customer cancels a pending/confirmed order → restock, refund prepaid to wallet, cancel delivery task | customer |
| `products:delete` | Seller/admin remove a listing (soft-delete → `removed`) | seller/admin |
| `wishlist:list` / `wishlist:toggle` | Persist the shopper wishlist server-side (new `WishlistItem` entity) | customer |
| `sellers:update` | Seller edits their own store profile (name/description) | seller |
| `reviews:helpful` | Mark a review helpful (new `helpful` counter on `Review`) | all |
| `notifications:markAllRead` | Clear the whole notification tray at once | all |
| `delivery:assign` | Ops assign an unassigned task to a specific rider | ops |

## 2. Unwired write-actions (handlers exist, UI does mock mutations)

These already have working handlers — they need the **frontend** click wired to
`socketClient.request(...)`. Tracked for the follow-up frontend pass:

- **Seller:** `products:update` (pause/unpublish), `campaigns:create`,
  `payouts:request` (partially wired), store settings save (`sellers:update`).
- **Admin:** `sellers:setStatus` (approve/reject/suspend), product moderation
  (`products:update`), `disputes:resolve/reject`, `customers:setActive`,
  `fraud:setStatus`, `roles:setRole`, `broadcasts:send`, `flashsales:create`,
  `promos:create`, `cms:upsert`, `settings:set`, `categories:create/update/remove`,
  `campaigns:setStatus`.
- **Warehouse:** `warehouse:buildBatch`, `warehouse:createTransfer`,
  `delivery:updateStatus` (advance parcel stage), `exceptions:create/setStatus`,
  `replacements:setStatus`, `exchanges:setStatus`.
- **Rider:** `delivery:accept`, `delivery:updateStatus`, `delivery:location`
  (map actions).

## 3. Verified sound (no action needed)

- Reads for every portal (products/orders/deliveries/payouts/disputes/analytics/
  customers/audit/fraud/cms/roles/campaigns/exceptions/…).
- Payments: intentional mock gateway (card/mobile-money always authorize) — by design.
- Auth, encryption, wallet, refunds, order→delivery pipeline.
