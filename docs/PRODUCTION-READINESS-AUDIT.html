Prod-grade

Already production-grade

  • Transport & CORS — REST locked to an allow-list; the Socket.IO adapter applies the same list. WebSocket-first, no polling.
  • Auth crypto — bcrypt (cost 12); JWT access+refresh with token revocation (tokenVersion).
  • Encryption at rest — AES-256-GCM on email/phone/address, with a deterministic emailHash for lookups.
  • Validation & limits — global ValidationPipe on REST; auth endpoints throttled (30/min/IP).
  • Coverage — 105 role-scoped handlers; end-to-end smoke passes 26/26; per-flow socket smokes pass.
  • Ops artifacts — Docker (API + web), compose, nginx, pm2, CI (build + seed + smoke), health endpoint.

P0 — Blockers

P0P0-1

Frontend auth is a demo shell — no real users

The login / register form ignores the credentials the user types and signs into a shared seeded account.

// web/src/components/shop/auth-form.tsx:23
login("customer");   // ignores entered email/password
// web/src/context/auth-context.tsx
DEMO_PASSWORD = process.env.NEXT_PUBLIC_DEMO_PASSWORD ?? "Somba@2026";

Every visitor becomes one of 7 demo accounts. The real backend auth (/auth/register|login|refresh) works, but the UI never calls it with real input — no registration, no per-user data, no real sessions.

FixWire auth-form.tsx and the portal entries to useRealtime().login/register; keep the persona picker behind a dev-only flag.
P0P0-2

DB_SYNCHRONIZE defaults to true — data-loss risk

// api/src/config/configuration.ts
synchronize: (process.env.DB_SYNCHRONIZE ?? 'true') === 'true'
// database.module.ts — applied to MySQL too

TypeORM synchronize auto-alters the live schema on boot — it can drop/rename columns and destroy data. There are no migrations.

FixDefault to false; generate TypeORM migrations; run them in the deploy step with DB_SYNCHRONIZE=false.
P0P0-3

Insecure default secrets, no fail-fast

// api/src/config/configuration.ts
jwt.secret        ?? 'dev-insecure-jwt-secret-change-me'
jwt.refreshSecret ?? 'dev-insecure-refresh-secret-change-me'
dataEncryptionKey ?? '0000…0000'   // all-zero 32-byte key

The app boots in production with these if env is unset — forgeable tokens and worthless encryption.

FixOn startup, if NODE_ENV=production and any secret is missing or equals its dev default, throw and refuse to boot.
P0P0-4

Payments are a mock gateway

// api/src/payments/payments.service.ts:49
// stripe_card / airtel_money — mock authorize (always succeeds)
status = 'succeeded';

Card and mobile-money charges always “succeed”; no real PSP, no idempotency keys, no webhook reconciliation. Fine for demo, unacceptable for real money.

FixIntegrate a real PSP (Stripe / Flutterwave / mobile-money) behind the existing PaymentsService seam; add idempotency + webhook confirmation before marking orders paid.

P1 — High

P1P1-1

No socket token refresh — users drop every 15 min

Access TTL is 15 min and the socket carries it in the handshake. On expiry the server rejects it → unauthorized → logout; reconnection retries with the same stale token.

FixOn unauthorized/connect_error, call /auth/refresh and reconnect with the new token; refresh proactively before expiry.
P1P1-2

WebSocket layer is unthrottled

ThrottlerGuard covers only the auth REST controller. Every read/write flows through the socket with no rate limiting — an authenticated client can flood any handler.

FixAdd per-socket, per-event rate limiting in the gateway (token-bucket keyed by user id).
P1P1-3

WebSocket payloads are unvalidated

Handlers use untyped @MessageBody(); the global ValidationPipe only acts on decorated DTO classes — none exist under src/realtime/. Only manual null checks, no type/enum/bounds validation.

FixDefine DTO classes per event (or validate with zod at each handler head) and apply validation to the gateway.
P1P1-4

No automated test suite (0 unit/integration tests)

Only the e2e smoke exists. Business logic — cancel/restock, refunds, payouts, disputes — has no regression net across 105 handlers.

FixUnit-test services (orders, payments, wallet, payouts) + integration-test critical socket flows; wire into CI.

P2 — Medium · P3 — Polish

P2

Hardening & resilience gaps

  • No helmet on REST — missing HSTS / CSP / X-Frame-Options headers.
  • No React error boundaries (0 app/**/error.tsx) — a render throw white-screens the route.
  • No graceful shutdown — enableShutdownHooks() not called; in-flight sockets dropped on deploy.
  • No structured logging / error monitoring — console.log only; no request ids, no Sentry/OTel.
  • SQLite is the default DB — production must set DB_TYPE=mysql with a connection pool.
P3

Polish

  • Hardcoded api/v1 prefix per controller — prefer setGlobalPrefix + versioning.
  • No OpenAPI/Swagger for REST; no documented socket-event contract.
  • No request-id / correlation-id propagation.
  • ~A dozen mock-only frontend write-actions + RMA/exception list pages remain (see docs/BACKEND-AUDIT.md) — feature-complete, not launch-blocking.
  • Product images reference remote Unsplash URLs (blocked by CSP in some envs) — move to owned/CDN assets.

Recommended order of work

  1. P0-3 — fail-fast on default secrets. One file, immediate safety.
  2. P0-2 — synchronize:false + generate migrations. Prevents data loss.
  3. P0-1 — wire real login / register. Unlocks real users.
  4. P1-1 — socket token refresh. Sessions survive past 15 minutes.
  5. P0-4 — real payment gateway. Largest external dependency; scope separately (needs a PSP + credentials).
  6. P1-2 / P1-3 — WebSocket throttle + payload validation, then P1-4 tests, then P2 / P3.
